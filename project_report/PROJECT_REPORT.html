Section 01
Problem Statement

In today's digital age, individuals and organizations need an accessible platform to publish, share, and discuss written content. Existing blogging platforms are either overly complex, proprietary with limited control, or lack modern UX design.

🎯 Why This System Is Needed
  • Content Publishing — Authors need a simple interface to create, edit, and delete posts with category organization and tagging
  • Reader Engagement — Readers need the ability to browse, search, and leave comments on posts
  • Secure Authentication — The platform must support user registration and login with JWT-based security
  • Modern Architecture — Demonstrates RESTful API design, client-server architecture, MVC pattern, and async programming
  • Educational Value — Showcases the complete MEAN stack as a practical demonstration of web engineering concepts
Section 02
System Architecture

DailyTea follows a three-tier client-server architecture with clear separation of concerns between the presentation, application, and data layers.

Presentation Layer

AngularJS 1.8 SPA

7 HTML templates, 4 controllers, 2 services, ngRoute routing, two-way data binding

Application Layer

Node.js + Express.js

RESTful API server, JWT authentication, input validation, error handling, middleware

Data Layer

MongoDB Atlas

Cloud-hosted NoSQL database, Mongoose ODM, schema validation, 3 collections

⟵ HTTP/REST (JSON) ⟶
🏗️ Key Architectural Patterns
  • MVC Pattern — Models (Mongoose schemas), Views (AngularJS templates), Controllers (Express + AngularJS controllers)
  • RESTful API Design — Standard HTTP methods (GET, POST, PUT, DELETE) with resource-based URIs
  • Client-Server Separation — AngularJS SPA served as static files, Express.js handles API only
  • Async Programming — All database operations use async/await with try/catch error handling
  • JWT Stateless Auth — JSON Web Tokens enable stateless authentication without server-side sessions

Section 04
Database Design

MongoDB Atlas stores data across three linked collections: Users, Posts, and Comments, connected through ObjectId references.

👤 User
_idObjectId · PK
usernameString · Unique
emailString · Unique
passwordString · Hashed
bioString · Optional
createdAtDate · Auto
📝 Post
_idObjectId · PK
titleString · Required
contentString · Required
excerptString · Auto
categoryString · Enum
tags[String]
authorObjectId · FK→User
createdAtDate · Auto
💬 Comment
_idObjectId · PK
bodyString · Required
postObjectId · FK→Post
authorObjectId · FK→User
createdAtDate · Auto
🔗 Relationships
  • User → Posts — One-to-Many (a user writes many posts)
  • User → Comments — One-to-Many (a user writes many comments)
  • Post → Comments — One-to-Many (a post has many comments)
  • Cascade Delete — Deleting a post removes all its comments
